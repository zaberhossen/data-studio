# Data Studio

A high-performance, browser-based **Data Analytics & BI platform** (Looker
Studio-style). The defining idea: all heavy data work вҖ” filtering, `GROUP BY`,
and aggregation over hundreds of thousands of rows вҖ” runs in **Rust compiled to
WebAssembly**, executing on a **Web Worker** background thread so the React UI
stays at a locked 60 FPS.

---

## Tech Stack

| Layer | Technology | Why |
|---|---|---|
| **Framework** | [Next.js 14](https://nextjs.org) (App Router) + TypeScript | File-based routing, RSC, first-class web-worker bundling |

| **Compute engine** | **Rust вҶ’ WebAssembly** via [`wasm-bindgen`](https://rustwasm.github.io/wasm-bindgen/) + [`serde`](https://serde.rs) | Near-native speed for filter/aggregate over large datasets |
| **Threading** | Native [Web Worker](https://developer.mozilla.org/docs/Web/API/Web_Workers_API) (`{ type: "module" }`) | Keeps the main thread free вҶ’ 60 FPS |
| **JS вҶ” Rust bridge** | [`serde-wasm-bindgen`](https://github.com/cloudflare/serde-wasm-bindgen) | Deserializes JS objects straight into Rust structs (no `JSON.stringify`) |
| **State** | React hooks (`useState`/`useRef`/`useMemo`) + a custom engine hook | Zustand is wired in `package.json` for future global state |
| **Build (WASM)** | [`wasm-pack`](https://github.com/rustwasm/wasm-pack) (`--target web`) | Emits the `.wasm` binary + JS glue into `src/wasm/pkg` |
| **Package manager** | pnpm | вҖ” |

## How the JS вҶ” Rust bridge works

The whole system is **one contract** вҖ” [`src/lib/types/analytics.ts`](src/lib/types/analytics.ts)
вҖ” expressed in three places (React, Worker, Rust). The data flow:

```
 в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ   load(rows) Г—1     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ   load_dataset()    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
 в”Ӯ React        в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә в”Ӯ  Web Worker  в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә в”Ӯ   Rust   в”Ӯ
 в”Ӯ Query panel   в”Ӯ  runQuery(query)   в”Ӯ chart.worker в”Ӯ   query()           в”Ӯ  (WASM)  в”Ӯ
 в”Ӯ shadcn/ui     в”Ӯ в—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Ӯ              в”Ӯ в—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Ӯ + DATASETв”Ӯ
 в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   ChartPayload       в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   ChartPayload      в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

1. **React** builds a declarative `Query` from shadcn `<Select>` controls and
   keeps the raw dataset in a `useRef` (never in state вҖ” React must never diff
   200k rows).
2. **The hook** ([`useAnalyticsEngine.ts`](src/hooks/useAnalyticsEngine.ts))
   spawns one worker and exposes a Promise API: `load(rows)` and
   `runQuery(query)`. Responses are matched to requests by `requestId`.
3. **The worker** ([`chart.worker.ts`](src/workers/chart.worker.ts)) lazily
   instantiates the `.wasm` binary, then calls the Rust exports.
4. **Rust** ([`lib.rs`](wasm/src/lib.rs)) deserializes rows **once** into a
   per-worker `thread_local` dataset, then runs
   `filter вҶ’ group_by вҶ’ aggregate вҶ’ sort вҶ’ limit` in a single pass and returns a
   tiny chart-ready `ChartPayload`.

**Key performance property:** the dataset crosses the thread boundary **once**;
every query thereafter ships only the small `Query` and returns ~50 aggregated
points. The main thread never iterates the raw rows вҶ’ 60 FPS.

### Supported query operations (in Rust)
- **Filters:** `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `contains`, `in_list`
- **Aggregations:** `SUM`, `AVG`, `COUNT`, `MIN`, `MAX`
- **Grouping:** `GROUP BY` any dimension column
- **Sort / limit:** ascending/descending by metric, top-N

## Getting Started

### Prerequisites
```bash
# Rust + WASM toolchain (one time)
rustup target add wasm32-unknown-unknown
cargo install wasm-pack
```

### Install & run
```bash
pnpm install
pnpm dev          # predev compiles Rust вҶ’ WASM, then starts Next.js
```
Open http://localhost:3000.

### Useful scripts
| Command | Action |
|---|---|
| `pnpm dev` | Build WASM (dev) + run Next.js dev server |
| `pnpm build` | Build WASM (release) + production Next.js build |
| `pnpm wasm:build` | Release WASM build only (`wasm-pack --release`) |
| `pnpm wasm:dev` | Dev WASM build only |
| `pnpm typecheck` | `tsc --noEmit` |
| `cd wasm && cargo test` | Native unit tests for the engine (no WASM needed) |

## Performance notes & roadmap

- **Dataset is loaded into the worker once** and parked in Rust memory; queries
  send only the query. This keeps per-query time low and constant (WASM linear
  memory never shrinks, so repeated full-dataset deserialization is avoided).
- **Next:** swap the row-map model for a **columnar / Apache Arrow** buffer for
  another large speedup; multi-source connectors (replace
  [`mock-source.ts`](src/lib/data/mock-source.ts) with a `/api/datasources`
  fetch вҖ” the `Row[]` shape downstream is unchanged); drag-and-drop dashboard
  grid with multiple chart widgets.
